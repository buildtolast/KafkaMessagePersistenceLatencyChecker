## Problem Statement

The relay-chain benchmark (producer-service, consumer-service, dashboard-service) surfaces rich operational data — Prometheus metrics per stage/path, `docker logs` per container — but nothing acts on it. Spotting a slow stage, a Mongo fetch that's crept up, or a code-quality issue still requires a human to notice the dashboard, go read the source, and manually write, test, build, and deploy a fix. There's no continuous, low-friction loop that watches the running system and turns "this looks off" into a reviewable, tested, deployable change.

## Solution

A new, separately-deployed `sre-agent-service` continuously watches the three app services' metrics and logs, correlates sustained timing regressions and static code-quality issues (via the existing `java-clean-code` rulebook) with the specific source files responsible, and — once a finding has accumulated enough confirmation — asks a local LLM to draft an actual fix as a local git branch with a real diff. Its own dashboard (separate port, separate UI from the read-only `dashboard-service`) shows every finding, its explanation, and its diff. Nothing is applied automatically: approval triggers a test-gated build of the affected service and a surgical restart of only that service's containers. The analysis loop paces itself adaptively so it doesn't hammer the local LLM when nothing new is happening.

## User Stories

1. As a benchmark operator, I want the SRE agent to continuously watch producer/consumer/dashboard-service's Prometheus metrics, so that timing regressions are caught without me staring at the dashboard.
2. As a benchmark operator, I want the SRE agent to use `docker logs` only as supporting evidence (errors, stack traces) rather than the primary timing signal, so that findings are grounded in the same accurate metrics the benchmark itself already trusts.
3. As a benchmark operator, I want a slow stage/method to be confirmed across 3 consecutive analysis cycles before it's proposed as a fix, so that a single transient blip (GC pause, one slow disk write) doesn't spawn a needless code change.
4. As a benchmark operator, I want static code-quality issues (java-clean-code rule violations) to surface on first detection with no confirmation wait, so that I don't wait around for a problem that was never going to fix itself with more observation.
5. As a benchmark operator, I want each finding mapped to the specific source file(s) responsible for that metric/stage, so that the LLM's context is targeted instead of dumping whole service trees on every analysis call.
6. As a benchmark operator, I want the analysis cadence to start fast (2 minutes) and back off adaptively (doubling, capped at 10 minutes) when nothing new is found, so that a quiet, healthy system doesn't keep hammering the local LLM for no reason.
7. As a benchmark operator, I want the cadence to reset to the fast interval the moment a new finding appears, so that a burst of related issues gets investigated promptly rather than at the slow end of the backoff.
8. As a benchmark operator, I want every finding — pending, observed-but-unconfirmed, and proposed — to persist to disk, so that restarting the SRE agent doesn't silently lose a finding I hadn't reviewed yet.
9. As a benchmark operator, I want a proposed fix to include the actual diff, not just a prose explanation, so that I can see exactly what code is about to change before I approve anything.
10. As a benchmark operator, I want a proposed fix to run the affected service's existing test suite before I can approve deployment, so that a red build is visible and blocks the fix rather than being applied blind.
11. As a benchmark operator, I want a failed test run on a proposed fix to be reported clearly on the dashboard (not silently retried or hidden), so that I understand why a fix isn't deployable as-is.
12. As a benchmark operator, I want approving a fix to merge it locally (branch → master) rather than opening a real GitHub PR, so that the loop stays fast and doesn't depend on network/GitHub Actions for a benchmark that only runs on my machine.
13. As a benchmark operator, I want approving a fix to trigger a surgical rebuild/restart of only the affected service's containers, so that fixing one service doesn't discard the in-flight relay-chain state in Kafka/Mongo or restart unrelated services.
14. As a benchmark operator, I want the SRE agent to run as its own containerized service with its own Docker socket and repo mount, so that only this one privileged container ever touches host-level git/build/Docker operations — the existing passive `dashboard-service` stays exactly as low-privilege as it is today.
15. As a benchmark operator, I want the SRE agent's dashboard on its own port, separate from the existing metrics dashboard, so that the two UIs' very different capability profiles (read-only viewer vs. privileged code-changing agent) stay visibly distinct.
16. As a benchmark operator, I want the SRE agent to bundle its own JDK21+Gradle rather than launching throwaway build containers, so that there's exactly one layer of container-launches-container complexity in the system (the socket-mounted agent itself), not two.
17. As a benchmark operator, I want the java-clean-code rulebook's severity/priority ordering and ID citations reused verbatim for quality findings, so that a fix's explanation reads consistently with the manual review I already did on this codebase.
18. As a benchmark operator, I want a rejected finding to be dismissible from the dashboard without deleting its history, so that I can see what was suggested and declined without re-triggering the same suggestion every cycle.
19. As a benchmark operator, I want the SRE agent's analysis engine to be a pure function I can test without Docker, git, or an LLM running, so that the adaptive cadence and confirmation-threshold logic is verified deterministically.
20. As a developer extending this system, I want MetricsSource, LogSource, LlmClient, CodeWorkflow, and ContainerOps to be small, single-purpose ports (mirroring PayloadReader/PayloadStore's existing shape in this codebase), so that each I/O boundary is independently fakeable in tests.
21. As a benchmark operator, I want the metric-to-class map to be explicit and reviewable (not inferred by the LLM), so that I can trust which files a given finding will actually touch before any LLM call is made.
22. As a benchmark operator, I want docker log rotation limits configured for the three app services, so that continuous log-tailing by the SRE agent doesn't choke on unbounded log growth over a long-running session.
23. As a benchmark operator, I want the SRE agent's own container never to be a target of its own findings in a way that causes it to modify or restart itself mid-analysis, so that the watcher doesn't destabilize itself.
24. As a benchmark operator, I want to see, per finding, which analysis cycle first observed it and how many consecutive cycles it's been confirmed for, so that I can judge how sustained a pattern is before deciding to approve.

## Implementation Decisions

- New service `sre-agent-service`, added to `docker-compose.yml`, with the repo directory bind-mounted in and `/var/run/docker.sock` mounted in for Docker CLI access. Own Dockerfile bundling JDK 21 + Gradle (matching the other three services' toolchain) so `./gradlew test` runs natively inside the container against the mounted repo.
- Own static dashboard UI + backend, structurally mirroring `dashboard-service`'s existing pattern (a small Spring Boot app serving a static page + SSE/poll endpoint), but on a separate port (suggested `8086`) and with no shared code/container with `dashboard-service`.
- Core pure seam: `AnalysisEngine.process(MetricsSnapshot current, FindingsHistory history, IntervalState interval) -> AnalysisResult` where `AnalysisResult` carries the updated history, the next interval delay, and the list of findings that just graduated to "proposed fix" this cycle. No I/O inside this function — it is constructed from data already fetched by the ports below.
- Adaptive interval: baseline 2 minutes; on any cycle producing zero new findings, the interval doubles, capped at 10 minutes; any cycle producing a new finding resets the interval to 2 minutes. Interval state is part of `IntervalState`, itself part of the persisted `FindingsHistory`.
- Confirmation rule: a timing-based finding (a stage/method whose latency is notably above its own rolling baseline) must appear in 3 consecutive analysis cycles before graduating from "observed" to "proposed fix." A code-quality finding (java-clean-code rule violation) graduates on the first cycle that detects it — no multi-cycle wait.
- Metric-to-class map: an explicit, reviewable lookup (config or small static table, not LLM-inferred) from metric name+tag combinations (e.g. `stage.hop.latency{stage=N,path=CLAIM_CHECK}`) to the specific class(es) responsible (e.g. `StageProcessor`, `MongoPayloadStore`, `MongoPayloadReader`), scoped per service. Used to select exactly which source files get sent to the LLM for a given finding — never the whole service tree.
- Signal sourcing: `MetricsSource` scrapes `/actuator/prometheus` from producer-service, consumer-service (all 3 replicas), and dashboard-service — the same endpoints `dashboard-service`'s own `ScrapeScheduler` already scrapes. `LogSource` fetches recent `docker logs` for the implicated service's container(s), filtered to ERROR-level and stack-trace lines, used only as supporting context alongside the metrics-derived finding, never as the primary timing signal.
- `LlmClient` port has two operations: one to analyze a graduated finding (metrics + mapped source files + supporting log lines in, a human-readable explanation + a confidence/rationale out) and one to generate an actual diff for an approved-for-drafting finding (same inputs, a unified diff out). Both call the existing local LLM at `127.0.0.1:8080` per this project's established convention; no new model/provider introduced.
- `CodeWorkflow` port: creates a local git branch, applies the LLM-generated diff as a commit, invokes `./gradlew test` for the affected service, and — only on your dashboard approval and only if tests passed — merges the branch into `master` locally. No GitHub push, no `gh pr create`, no GitHub Actions.
- `ContainerOps` port: on successful local merge, runs `docker compose build <service>` then `docker compose up -d --no-deps <service and its replicas>` for just the affected service — Kafka/Mongo/other app services are left running and untouched.
- Persistence: `FindingsHistory` (all findings — observed, confirmed, proposed, approved, rejected — plus current `IntervalState`) is serialized to a JSON file on a volume mounted into the `sre-agent-service` container, so it survives that container's own restarts.
- Existing docker-compose services (producer, consumer1-3, dashboard) get log rotation limits added (`logging: driver: json-file, options: max-size/max-file`) so continuous log-tailing has a bounded volume to work against.
- The SRE agent explicitly excludes its own service from its metric/log watch scope and from the metric-to-class map — it never proposes a fix to itself.
- Dashboard interactions: each finding shows its first-observed cycle, confirmation count (for timing findings), explanation, and (once graduated) diff. Actions available: Approve (triggers CodeWorkflow → ContainerOps), Reject (persists as rejected, does not re-surface the identical finding on subsequent cycles unless the underlying metric/code changes again).

## Testing Decisions

- `AnalysisEngine` is the primary unit-test target: pure function, no Docker/git/LLM required, following the exact TDD pattern already used for `StageProcessor` (RED-first tests against fakes/inputs, then implementation). Test cases should cover: interval backoff progression (2→4→8→10, capped), interval reset on new finding, 3-cycle confirmation gate for timing findings, first-cycle graduation for quality findings, and no re-surfacing of a rejected finding.
- `MetricsSource`, `LogSource`, `LlmClient`, `CodeWorkflow`, and `ContainerOps` are each tested against fakes at their interface boundary, mirroring `StageProcessorTest`'s `FakePayloadReader`/`FakePayloadStore` pattern — no real Docker socket, git repo, or LLM call in unit tests.
- Integration-level testing (if pursued) would use Testcontainers-style real interactions only for the parts genuinely worth it (e.g. a real `git` repo in a temp directory for `CodeWorkflow`), following this codebase's existing Testcontainers precedent for Kafka/Mongo — not for the LLM calls, which stay faked even in integration tests given their non-determinism.
- The metric-to-class map itself is a plain data structure and should have a small direct test confirming each known metric name+tag combination resolves to the expected class list.

## Out of Scope

- Any change to producer/consumer/dashboard-service's actual application logic as part of this spec — this spec is only the SRE agent infrastructure itself; the fixes it eventually proposes are separate, future, per-finding changes.
- Real GitHub PRs, GitHub Actions, or any remote CI — explicitly decided against in favor of the local PR-style workflow.
- Security/dependency/test-coverage scanning — scope is strictly the java-clean-code rulebook plus timing regressions, per the explicit decision to keep security as a separate, deliberate workflow.
- Kafka/Mongo infrastructure log analysis — app-service logs only.
- A host-level (non-containerized) deployment mode — explicitly decided in favor of containerized with a mounted Docker socket.
- Batched multi-finding approval — each finding is approved and deployed independently; no mechanism in this spec for combining multiple pending fixes into one merge/restart cycle.
- Automatic rollback of a deployed fix if it regresses metrics after restart — this spec covers propose→approve→deploy, not post-deploy monitoring/rollback of the SRE agent's own changes.

## Further Notes

- This is a genuinely separate context from the relay-chain benchmark itself (see CONTEXT.md's "SRE Agent" section, added this session) — it operates on the benchmark's code and containers but isn't part of the benchmark's own architecture.
- The Docker socket mount is a well-understood but real security tradeoff (effectively root-equivalent host access) — worth the team being explicitly aware of it, even though it was a deliberate, discussed choice rather than an oversight.
- Given the adaptive interval and 3-cycle confirmation gate compound (worst case ~30-40 minutes from first observation to proposed fix at the slow end of the backoff), this is intentionally a "continuously improving over hours/days" tool, not a fast-feedback one — consistent with the stated goal of "keep improving and making suggestions" over time rather than instant reaction.
- Implementation routing should follow this project's established CLAUDE.md §12 convention: local LLM for codegen, Sonnet-tier planning/specs, review before acceptance — applies to building the SRE agent's own code, not to be confused with the SRE agent's own use of the local LLM at runtime for its findings.
